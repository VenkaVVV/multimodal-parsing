# 文档解析预处理方案对比分析报告

## 一、背景

本报告对比两套文档预处理方案：**现有方案**（基于华为内部设计，Page 28-31）和**推荐方案**（基于业界最新开源实践），从架构思路、工具选型、性能、可维护性、RAG 适配度等维度进行系统性评估，为最终技术选型提供依据。

---

## 二、现有方案概述

现有方案采用**按文件格式独立设计 Pipeline**的思路，每种格式有专属的解析路径：

### 2.1 Word 格式（Page 28）

| 步骤 | 处理逻辑 |
|------|----------|
| 格式统一 | doc → LibreOffice → docx |
| 结构解析 | docx → mammoth → HTML |
| 内容提取 | HTML → BeautifulSoup → JSON |

核心设计理由：docx 先转 HTML 再解析，速度远优于直接解析 docx（2 万篇从 24 小时降至 3 分钟）。HTML 作为中间格式，后续可灵活调整分片逻辑。

### 2.2 Excel 格式（Page 29）

| 步骤 | 处理逻辑 |
|------|----------|
| 格式统一 | xls → calamine → xlsx |
| 按 Sheet 解析 | xlsx → calamine → sheetN |
| 表格转换 | Sheet → HTML 表格 → BeautifulSoup → Markdown 表格 |

核心设计理由：前台页面预览 HTML 格式效果更好；Sheet 也可直接转 Markdown，检索和模型调用更方便。

### 2.3 PDF 格式（Page 30）

| 步骤 | 处理逻辑 |
|------|----------|
| 解析 | PDF → Fitz (PyMuPDF) → page |
| 快照 | 每页生成 (doc_id)\_(part_id)\_(aid).png |

核心设计理由：针对 PDF 中的图片存在解析能力不足的问题，通过页码将检索到的知识与快照关联，展示给用户更直观。

### 2.4 PPT 格式（Page 31）

| 步骤 | 处理逻辑 |
|------|----------|
| 格式统一 | ppt → LibreOffice → pptx |
| 双路径解析 | 路径 A：pptx → python-pptx → Slide → 拼接文本内容 |
|  | 路径 B：pptx → LibreOffice → PDF → Fitz → 每页快照 |

核心设计理由：python-pptx 提取结构化文本；同时生成每页快照用于前端展示，弥补 PPT 图片解析能力不足的问题。

## 三、推荐方案概述

推荐方案采用**统一管线 + 版面检测分区处理**的思路，所有格式归一化后走统一的 AI 识别管线：

### 3.1 整体流程

| 阶段 | 处理逻辑 |
|------|----------|
| 格式归一化 | 所有格式 → 页面图像 + 原生文本提取（数字化 PDF 可同时获取） |
| 版面检测 | DocLayout-YOLO 检测每页的文本/表格/图片/标题/公式区域 |
| 阅读顺序 | XY-Cut++ 算法确定区域的正确阅读顺序 |
| 分区识别 | 文本区 → PaddleOCR；表格区 → StructTable/TableFormer；公式区 → UniMERNet |
| 双模式 | Mode A（传统小模型管线）/ Mode B（VLM 大模型逐区域识别） |
| 结构整合 | 按阅读顺序 + 标题层级拼装完整 Markdown |
| 语义分片 | 结构感知切分 + 语义边界精调 |

### 3.2 核心工具选型

| 环节 | 工具 |
|------|------|
| 格式转换 | LibreOffice headless |
| PDF 解析 | PyMuPDF (Fitz) |
| 版面检测 | DocLayout-YOLO（CVPR 2025 评测 SOTA） |
| OCR | PaddleOCR v5 |
| 表格识别 | StructTable-InternVL2-1B |
| 公式识别 | UniMERNet |
| VLM | DeepSeek-VL2 / Qwen2.5-VL |
| 整合框架 | MinerU（二次开发） |

## 四、逐格式深度对比

### 4.1 Word 格式对比

| 维度 | 现有方案（mammoth → HTML → BS4） | 推荐方案（统一管线） |
|------|------|------|
| 结构保真度 | ★★★★☆ mammoth 能较好保留标题、列表、表格的 HTML 语义结构 | ★★★☆☆ 转为图片后依赖 OCR 重建结构，会有信息损失 |
| 复杂表格 | ★★★★☆ 原生表格结构直接转 HTML，合并单元格保留完好 | ★★★☆☆ 表格区域需要 StructTable 重新识别，复杂合并单元格可能出错 |
| 嵌入图片处理 | ★★☆☆☆ mammoth 可提取图片但无法理解图片内容 | ★★★★☆ VLM 模式可理解图片内容并生成描述 |
| 处理速度 | ★★★★★ 纯文本操作，极快（2 万篇 3 分钟） | ★★☆☆☆ 每页需模型推理，速度慢 1-2 个数量级 |
| 中间格式价值 | ★★★★☆ HTML 中间格式灵活，可复用于分片 | ★★★☆☆ 直接输出 Markdown，无中间格式 |

**结论：Word 格式现有方案更优。** mammoth → HTML 的路径在结构保真度和速度上都有明显优势。docx 本身是结构化的 XML 格式，没有必要转图片再做 OCR——那是对已有结构信息的浪费。

**改进建议：** 在现有 mammoth 管线基础上，对 HTML 中的 `<img>` 标签增加 VLM 图片理解，为嵌入图片生成文本描述后插入 Markdown，实现两种方案的优势互补。

### 4.2 Excel 格式对比

| 维度 | 现有方案（calamine → HTML → MD） | 推荐方案（统一管线） |
|------|------|------|
| 数据精度 | ★★★★★ 直接读取单元格数据，零损失 | ★★☆☆☆ OCR/VLM 识别数字可能出错（0.1M vs 0.lM） |
| 合并单元格 | ★★★★☆ calamine + BS4 能正确处理 | ★★★☆☆ 依赖表格识别模型重建 |
| 多 Sheet 支持 | ★★★★★ 原生支持 | ★☆☆☆☆ 转图片后 Sheet 信息丢失 |
| 公式/数据类型 | ★★★★☆ 可保留公式计算结果、日期等 | ★☆☆☆☆ 无法理解 |
| 处理速度 | ★★★★★ 纯数据读取，毫秒级 | ★★☆☆☆ 需模型推理 |

**结论：Excel 格式现有方案绝对更优。** Excel 是纯结构化数据格式，用 calamine 直接解析是最正确的做法。将 Excel 转图片再 OCR 是降维操作，不仅慢而且会引入数字识别错误，在金融场景下不可接受。

**改进建议：** calamine 方案基本无需改进。建议增加一个判断：对于包含图表（chart）的 Sheet，额外走截图 + VLM 理解的路径来提取图表语义信息。另外可以考虑用 openpyxl 替代 calamine 获取更丰富的格式信息（如单元格样式、条件格式等），但 calamine 在速度上更有优势。

### 4.3 PDF 格式对比

| 维度 | 现有方案（Fitz → page → 快照） | 推荐方案（统一管线） |
|------|------|------|
| 文本提取 | ★★☆☆☆ Fitz 提取原生文本但不保留结构 | ★★★★★ 版面检测 + 分区识别，结构完整 |
| 表格识别 | ★☆☆☆☆ 无专门的表格识别 | ★★★★☆ StructTable/TableFormer 专项处理 |
| 扫描件处理 | ★☆☆☆☆ 依赖 Fitz 原生文本，扫描件无法处理 | ★★★★★ OCR + 版面检测完整覆盖 |
| 复杂版式 | ★★☆☆☆ 多栏、混合版式下文本顺序混乱 | ★★★★☆ 阅读顺序检测解决多栏问题 |
| 快照关联 | ★★★★★ 每页快照可直接展示给用户 | ★★☆☆☆ 需额外生成快照 |
| 公式识别 | ☆☆☆☆☆ 无 | ★★★★☆ UniMERNet 转 LaTeX |

**结论：PDF 格式推荐方案显著更优。** 现有方案在 PDF 处理上过于简单——仅做了文本提取和快照生成，缺少版面理解、表格识别、扫描件 OCR 等核心能力。这是当前方案中最大的短板。

**改进建议：** PDF 格式应完整采用推荐方案的管线（DocLayout-YOLO → 分区识别 → Markdown 整合），同时保留现有方案中"每页快照"的设计思路用于前端展示。具体来说：

1. 数字化 PDF：Fitz 提取原生文本 + 坐标 → DocLayout-YOLO 检测版面 → 原生文本与检测框匹配 → 表格区走 StructTable
2. 扫描件 PDF：每页渲染为图片 → DocLayout-YOLO → 各区域走 OCR/表格识别
3. 快照功能保留：所有 PDF 页面都生成快照，RAG 检索时关联页码快照

### 4.4 PPT 格式对比

| 维度 | 现有方案（python-pptx + 快照双路径） | 推荐方案（统一管线） |
|------|------|------|
| 文本提取 | ★★★★☆ python-pptx 直接读取文本框内容 | ★★★☆☆ OCR 可能引入错误 |
| 版式理解 | ★★☆☆☆ 只拼接文本，丢失空间布局信息 | ★★★★☆ 版面检测保留布局关系 |
| SmartArt/图表 | ★☆☆☆☆ python-pptx 无法解析 | ★★★★☆ VLM 可理解图表语义 |
| 动画/过渡 | ★☆☆☆☆ 无法处理 | ★☆☆☆☆ 同样无法处理 |
| 快照展示 | ★★★★★ 完善的快照生成与关联机制 | ★★☆☆☆ 需额外处理 |
| 处理速度 | ★★★★☆ python-pptx 快，LibreOffice 转换有开销 | ★★☆☆☆ 每页需模型推理 |

**结论：PPT 格式建议融合两种方案。** 现有方案的 python-pptx 文本提取 + 快照的思路是对的，但在 SmartArt、图表、复杂版式上能力不足。推荐方案对这些场景有优势但速度慢。

**改进建议：**

1. 保留 python-pptx 的文本提取作为"快速通道"
2. 每页同时生成快照（保留现有设计）
3. 对于 python-pptx 无法提取的内容（SmartArt、图表、复杂图片），增加 VLM 补充识别路径
4. 具体判断逻辑：如果某页 python-pptx 提取到的文本量明显少于该页图片面积所预期的文本量，则触发 VLM 补充识别

## 五、架构层面对比

### 5.1 架构设计理念

| 维度 | 现有方案 | 推荐方案 |
|------|------|------|
| 设计理念 | 格式专用管线，各格式独立处理 | 统一管线，所有格式归一化处理 |
| 代码复用 | 较低，每种格式的工具链独立 | 较高，共用版面检测和分区识别模块 |
| 维护成本 | 各管线独立维护，改一个不影响其他 | 统一管线修改可能影响所有格式 |
| 扩展性 | 新增格式需从头设计管线 | 新增格式只需添加格式归一化模块 |

### 5.2 快照设计的独特价值

现有方案中**"每页快照"** 的设计是一个非常有价值的思路，推荐方案中缺少这一考量。在 RAG 场景下，快照的价值体现在：

1. **用户体验**：用户检索到某段知识后，可以看到该段知识在原始文档中的页面截图，直观确认上下文
2. **来源可追溯**：快照命名为 `(doc_id)_(part_id)_(aid).png`，建立了知识块与原始页面的精确映射
3. **弥补解析损失**：当解析结果有遗漏时，用户仍可通过快照获取完整信息

**建议：** 无论最终采用哪种管线，快照机制都应作为标准功能保留。

### 5.3 中间格式策略

现有方案的一个核心洞察是**将 docx 先转 HTML 再处理的中间格式策略**。这个策略的好处是：

1. HTML 是通用的结构化格式，后续的分片、转 Markdown、转 JSON 都很方便
2. BeautifulSoup 生态成熟，解析稳定
3. 变更分片策略时不需要重新解析原文件，直接从 HTML 中间格式出发即可

推荐方案直接从版面检测到 Markdown，缺少这种中间缓冲层。建议在最终方案中保留 HTML 作为中间格式的思路。

## 七、最终建议：融合方案

综合分析后，建议采用**"格式感知的分层处理架构"**——对结构化格式保留现有方案的直接解析路径（快且准），对非结构化内容引入推荐方案的 AI 管线（能力强），两者互补。

### 7.1 融合架构设计

```
文件输入
  │
  ├── 结构化格式（可直接解析原生结构）
  │     ├── Word (docx)  → mammoth → HTML → Markdown/JSON
  │     ├── Excel (xlsx) → calamine → HTML表格 → Markdown
  │     └── PPT (pptx)   → python-pptx → 文本提取 + 快照
  │           └── 补充：复杂页面 → VLM 识别
  │
  ├── 非结构化格式（必须走 AI 识别管线）
  │     ├── 扫描件 PDF → DocLayout-YOLO → 分区识别 → Markdown
  │     └── 图片      → DocLayout-YOLO → 分区识别 → Markdown
  │
  ├── 数字化 PDF（两种路径择优）
  │     ├── 快速模式 → Fitz 原生文本 + 坐标 → 表格区走 StructTable
  │     └── 精准模式 → DocLayout-YOLO 完整管线
  │
  └── 所有格式通用
        ├── 生成每页快照（doc_id_part_id_aid.png）
        ├── 输出 HTML 中间格式（便于后续复用）
        └── HTML → Markdown → 语义分片 → 向量化
```

### 7.2 关键设计原则

1. **能用结构化解析就不走 OCR**：Word/Excel/PPT 已有完善的结构化解析库，没必要转图片再 OCR
2. **AI 管线用于弥补短板**：对于 python-pptx 提取不到的内容、扫描件 PDF、复杂版式的数字 PDF，才启用 AI 管线
3. **快照作为标配**：所有格式都生成页级快照，用于前端溯源展示
4. **HTML 作为中间格式**：所有解析结果先统一为 HTML，再转 Markdown，保留中间缓冲层
5. **双模式可选**：AI 管线内部保留"传统小模型"和"VLM 大模型"两种模式供用户选择

### 7.3 融合方案的预期优势

| 维度 | 相比现有方案提升 | 相比纯推荐方案提升 |
|------|------|------|
| PDF 解析 | 大幅提升（从无到有的 AI 管线） | 保留快照机制 |
| 扫描件 | 大幅提升（新增 OCR 能力） | 无变化 |
| Word/Excel | 保持不变（已是最优路径） | 避免不必要的 OCR 降级 |
| 速度 | 结构化格式保持原有速度 | 结构化格式快 10-50 倍 |
| 部署复杂度 | 增加 GPU 需求（仅 PDF/扫描件需要） | 结构化格式不依赖 GPU |
